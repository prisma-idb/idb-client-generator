# Phase 7.7 — Migrate `apps/prisma-next-usage` to the new design

**Status**: Not started
**Depends on**: 7.1–7.5 (full stack must be functional)
**Blocks**: 7.8 (closure can't finish until the app is green)

## Goal

Convert the demo app from the old manifest-driven flow to the new
`ContractSpace`-driven flow. This is the **end-to-end** validation that
phases 7.1–7.5 actually work together.

After this phase:

- `apps/prisma-next-usage/` has a `migrations/app/<ts>_baseline/`
  package containing the current contract as the first migration.
- `apps/prisma-next-usage/src/lib/prisma/contract-space.generated.ts`
  exists and is regenerated by `prisma-next-idb generate-contract-space`.
- `apps/prisma-next-usage/prisma-idb.manifest.json` is **deleted**.
- `db.ts` consumes `contractSpace` instead of `contract` + manifest.
- `prisma-next.config.ts` no longer mentions the manifest driver.
- Playwright tests pass.
- (After 7.6 ships) `prisma-next-idb preflight` returns 0.

## Steps (intended commit order)

### Step 1 — Create the baseline migration package

Run inside `apps/prisma-next-usage/`:

```bash
pnpm prisma-next migration new --name baseline
```

This invokes the framework's `migration new` (already in
`prisma-next@0.11.0`), which calls our planner's
`emptyMigration({ packageDir, contractJsonPath, fromHash: null, toHash: <current contract storageHash> }, "app")`,
and writes the class-based `migration.ts` from 7.2 to
`migrations/app/<timestamp>_baseline/`.

Edit `migrations/app/<ts>_baseline/migration.ts` and replace its empty
`operations` with the actual ops the planner derives from the current
contract (`createObjectStoreOp` for `users` + `posts`, `createIndexOp`
for `byAuthorId`, `byEmail`, `byScore`).

**Alternative** (faster if framework's `migration new` isn't fully
wired for IDB yet): hand-author the `migration.ts` based on the current
`prisma-idb.manifest.json` schema:

```ts
#!/usr/bin/env -S npx tsx
import { Migration, MigrationCLI, createIndexOp, createObjectStoreOp } from "@prisma-next-idb/target-idb/migration";

export default class M extends Migration {
  override describe() {
    return {
      from: null,
      to: "sha256:c1ad80d3bd2f9f5db48c5126fe1257ba9817ddc6ba0a8af9453e39f17d76dd30",
    };
  }

  override get operations() {
    return [
      createObjectStoreOp("users", { keyPath: "id" }),
      createIndexOp("users", "byEmail", { keyPath: "email", unique: true }),
      createIndexOp("users", "byScore", { keyPath: "score", unique: false }),
      createObjectStoreOp("posts", { keyPath: "id" }),
      createIndexOp("posts", "byAuthorId", { keyPath: "authorId", unique: false }),
    ];
  }
}

MigrationCLI.run(import.meta.url, M);
```

Then run:

```bash
node migrations/app/<ts>_baseline/migration.ts
```

→ writes `ops.json` + `migration.json` next to the file.

Copy the current contract artifacts as the package's end-contract:

```bash
cp src/lib/prisma/contract.json     migrations/app/<ts>_baseline/end-contract.json
cp src/lib/prisma/contract.d.ts     migrations/app/<ts>_baseline/end-contract.d.ts
```

### Step 2 — Generate the contract-space module

```bash
npx prisma-next-idb generate-contract-space
```

→ writes `src/lib/prisma/contract-space.generated.ts` + `migrations/refs/head.json`.

Add to `.prettierignore` or whatever formatter ignore file the app uses:

```
migrations/refs/head.json
src/lib/prisma/contract-space.generated.ts
```

### Step 3 — Update `db.ts`

```ts
// apps/prisma-next-usage/src/lib/prisma/db.ts
import { createAutoMigratingIdbClient } from "@prisma-next-idb/client-idb/client-auto";
import { contractSpace } from "./contract-space.generated";

const DEFAULT_DB_NAME = "prisma-next-usage";
type IdbClient = Awaited<ReturnType<typeof createAutoMigratingIdbClient>>;

let _client: IdbClient | null = null;
let _clientDbName: string | null = null;

export function resolveDbName(): string {
  if (typeof window === "undefined") return DEFAULT_DB_NAME;
  const param = new URLSearchParams(window.location.search).get("db");
  return param && param.length > 0 ? param : DEFAULT_DB_NAME;
}

export async function getDb(): Promise<IdbClient> {
  const dbName = resolveDbName();
  if (_client && _clientDbName === dbName) return _client;
  if (_client) await _client.close();
  _client = await createAutoMigratingIdbClient({ contractSpace, dbName });
  _clientDbName = dbName;
  return _client;
}

export async function resetDb(): Promise<void> {
  // unchanged
}
```

Drop the `contract` parameter and the `manifest` parameter from `getDb`.
Update every caller in `src/` and `tests/`.

### Step 4 — Update `prisma-next.config.ts`

```ts
import { defineConfig } from "@prisma-next-idb/family-idb/config-types";
import { typescriptContract } from "@prisma-next-idb/family-idb/config-types";
import idbFamily from "@prisma-next-idb/family-idb/control";
import idbTarget from "@prisma-next-idb/target-idb/control";
import idbAdapter from "@prisma-next-idb/adapter-idb/control";
import { contract } from "./src/lib/prisma/contract.server";

export default defineConfig({
  family: idbFamily,
  target: idbTarget,
  adapter: idbAdapter,
  // No driver: there's no manifest to read/write from the CLI.
  // The framework's CLI commands that need a driver (db sign, db verify,
  // db update) will refuse with a structured IDB-target error.
  contract: typescriptContract(contract, "src/lib/prisma/contract.json"),
  migrations: {
    dir: "migrations",
  },
});
```

If `defineConfig` requires `driver` and `db.connection` as non-optional,
either:

- Make them optional in `family-idb/src/exports/config-types.ts` (we own
  this type), or
- Pass a no-op driver placeholder that just throws on any call.

Decide during implementation. Optional is cleaner.

### Step 5 — Delete the manifest

```bash
rm apps/prisma-next-usage/prisma-idb.manifest.json
```

Update `.gitignore` if it explicitly excludes/includes it (probably
neither — it's just tracked).

### Step 6 — Run the test suites

```bash
pnpm -F prisma-next-usage build
pnpm -F prisma-next-usage test:packages  # unit/playwright
```

Expect failures, fix them.

**Anticipated breakage**:

- Tests that import `ManifestLike` or `IdbManifestControlDriver` from
  family-idb. Remove or rewrite.
- Tests that read `prisma-idb.manifest.json` from disk. Remove or
  replace with `contractSpace.headRef.hash` lookups.
- Tests that call `prisma-next db sign` / `db verify` / `db update`. These
  now return refusal errors; update tests to assert the refusal.

### Step 7 — Run preflight

```bash
npx prisma-next-idb preflight
```

→ should exit 0 (baseline migration applies cleanly).

## Risks

1. **Existing IDB databases in browsers** that ran the old code have
   `{ id: "default", ... }` markers. The runtime legacy fallback in 7.4
   covers them. Spot-check by manually loading the old app, then
   loading the new app, in the same browser, with the same dbName.
2. **`with { type: 'json' }` import attributes**: Vite supports them
   since v5; we're already on Vite 5+ via SvelteKit. Worth a quick
   build smoke test before committing.
3. **`#!/usr/bin/env -S npx tsx`** shebang: the user needs `tsx`
   available (e.g. via `pnpm dlx tsx` or as a dev dep). Add `tsx` to
   `apps/prisma-next-usage/package.json` devDependencies just so the
   shebang's `npx tsx` resolves without a network round-trip.
